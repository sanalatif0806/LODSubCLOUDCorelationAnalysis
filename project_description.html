FAIR Assessment
Pipeline

An automated pipeline for evaluating the FAIRness of Knowledge Graphs drawn from the LOD Cloud, using two independent tools: F-UJI and FAIR-Checker.

5 modules 2 assessment tools LOD Cloud source CSV + JSON output

Overview

What this project does

This project measures how Findable, Accessible, Interoperable, and Reusable (FAIR) a curated set of Knowledge Graph datasets are. It starts from a structured catalogue of LOD Cloud datasets, groups them into topic-based sub-clouds, and then runs every dataset through two independent FAIRness evaluation services — producing per-topic score tables ready for further analysis.

The pipeline is designed for reproducible, large-scale batch assessment: once configured, a single command triggers the full workflow end-to-end.

🗂

LOD Cloud as source

Datasets are drawn from the Linked Open Data Cloud catalogue, partitioned into topical sub-clouds (e.g. life sciences, geography, government) for structured comparison.

📐

Dual FAIR evaluation

Each dataset is assessed independently by F-UJI (a local REST server) and FAIR-Checker (a remote API), providing two complementary FAIRness scores per dataset.

🔗

Baseline integration

A pre-existing KGHBeat FAIR baseline CSV is merged with the sub-cloud data, so every result row is enriched with prior expert annotations.

📊

Structured outputs

Results are written as per-topic CSV files — one file per tool per topic — making it easy to load directly into pandas, R, or any spreadsheet for analysis.

Pipeline

Five-step execution order

Each step is an independent Python module. The main.py entrypoint runs them in sequence, with optional step selection and error handling.

01
Generate per-topic CSV files LODsubcloudclasscsv.py

Reads kgs_by_topic.json — a mapping of topic names to LOD Cloud dataset URLs — and writes one <topic>.csv per topic into data/LODsubclouds/. Each CSV contains the extracted dataset IDs (the last path segment of each URL).

02
Generate per-topic JSON files LODsubcloudclassjson.py

Performs the same partitioning as Step 1, but writes structured JSON files (with topic, count, and id list) into a separate output directory, plus a _index.json summary of all topic sizes.

03
Merge with FAIR baseline mergefairscore.py

Left-joins each sub-cloud CSV against the KGHBeat expert assessment baseline (a wide CSV keyed on KG ID). The merged files, one per topic, are saved to FAIRASSout/. Datasets not present in the baseline appear with NaN columns, preserving the full sub-cloud.

04
F-UJI automated assessment fujiass.py

For every dataset ID in every sub-cloud CSV, constructs the LOD Cloud landing URL and sends a POST request to a locally running F-UJI server (localhost:1071). Flattens the nested JSON response — summary scores, per-metric scores, and per-test scores — into a single wide CSV row. Outputs one <topic>.csv and <topic>_failed.csv per topic.

05
FAIR-Checker assessment FAIR-Checkerass.py

Calls the public FAIR-Checker REST API for each dataset URL. Applies URL pre-processing (Bio2RDF redirects, BioPortal path rewriting, blocked-domain filtering) before the request. Aggregates the returned per-metric scores into principle totals (F / A / I / R) and an overall score. Results and failures are saved per topic.

Assessment tools

Two independent evaluation frameworks

Using two separate tools guards against tool-specific blind spots and makes score comparisons possible — a dataset that scores well on both frameworks has stronger FAIR evidence than one that scores well on only one.

F-UJI (FAIRsFAIR)

A self-hosted Python REST service that tests 17+ FAIR metrics developed by the FAIRsFAIR project. Produces granular per-metric and per-test scores across the full F-A-I-R spectrum. Requires a running local server instance. Access via localhost:1071.

FAIR-Checker

A publicly accessible API hosted by France Bioinformatique. Applies its own metric suite and returns per-metric scores with recommendations and comments. No local installation required — assessed over HTTPS with a configurable timeout and delay.

Structure

Repository layout

All source code lives under src/. Data inputs and outputs are written under data/ at the project root, keeping code and data cleanly separated.

Outputs

What the pipeline produces

  • CSV
    Per-topic F-UJI results — one wide CSV per topic containing flattened F-UJI scores: summary totals, per-metric earned/total/maturity, and per-test status for all 17+ metrics.
  • CSV
    Per-topic FAIR-Checker results — one CSV per topic with individual metric scores, per-principle aggregates (F / A / I / R total and average), overall score, and API-returned recommendations.
  • CSV
    Per-topic failure logs — both tools write a companion *_failed.csv recording which datasets could not be assessed and the reason (timeout, HTTP error, URL filtered, etc.).
  • CSV
    Baseline-merged sub-clouds — the LOD sub-cloud CSVs left-joined with the KGHBeat expert annotation baseline, giving every row a full context of prior FAIR scores alongside the automated results.
  • JSON
    Structured topic files — per-topic JSON objects with id lists and counts, plus a _index.json summary — useful for downstream tooling or dashboards.

Usage

Running the pipeline

Install dependencies once, then run main.py. A local F-UJI server must be running before Step 4 executes; the script will exit early with a clear message if it is not reachable.

# 1. install dependencies pip install -r requirements.txt # 2. start the F-UJI server (required for step 4) python -m fuji_server -c fuji_server/config/server.ini # 3. run the full pipeline python main.py # — or run specific steps only — python main.py --steps 1,2 # generate subcloud files only python main.py --steps 4 # F-UJI assessment only python main.py --stop-on-error # abort if any step fails
